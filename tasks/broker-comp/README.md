# Отчёт: сравнение RabbitMQ и Redis Streams

Стенд: Docker — RabbitMQ 3.13 и Redis 7.2 (Streams), по 2 vCPU / 1 GiB RAM на контейнер брокера; 1 producer : 1 consumer; целевые скорости 1000 / 5000 / 10 000 msg/s; размеры полезной нагрузки 128 B, 1 KiB, 10 KiB, ~100 KiB (сырой payload в JSON/base64).  
Длительность одного прогона: 22 с.

---

## 1. Цель и вопросы, на которые отвечает бенчмарк

Нужно было сопоставить два брокера в близких по смыслу режимах надёжности:

| Режим            | RabbitMQ                                        | Redis Streams               |
| ---------------- | ----------------------------------------------- | --------------------------- |
| `durable_ack`    | durable-очередь, publisher confirms, ручной ack | AOF + чтение группой + XACK |
| `inmemory_noack` | transient-очередь, без confirms (скорость)      | без AOF, чтение без XACK    |

Измерялись:

- устойчивая пропускная способность (`throughput_msg_s`, msg/s, после warmup);
- p95 задержки по телу сообщения (`send_ts` → приём на consumer);
- доля потерь в отчёте: `lost_count / sent_count × 100` (разрывы по `seq` на consumer).

## 4. Анализ результатов по сценариям

### 4.1. RabbitMQ `durable_ack`

- Throughput: стабильно ~165–266 msg/s в зависимости от размера; до целевых 1k–10k msg/s в этом стенде не дотягивает — узкое место: связка подтверждений, дисковая/персистентная очередь, лимиты CPU/RAM и один consumer.
- Потери: `sent_count = recv_count` во всех строках — ничего не «пропало» между producer и consumer.
- p95: на малых и средних сообщениях ~1,3–2,0 мс; на ~100 KiB p95 ~3,5–3,7 мс — рост ожидаем из‑за объёма тела и стоимости обработки.

**Итог:** режим предсказуем хорош по доставке, но по msg/s сильно проигрывает

---

### 4.2. Redis Streams `durable_ack`

- На 128 B–10 KiB при 5k и особенно 10k цели достигаются тысячи msg/s (пики порядка ~7,9k msg/s на 128 B при 5k цели).
- На ~100 KiB для всех целевых скоростей в данных зафиксировано одинаковое `sent_count = recv_count = 5750` за 22 с → фактически ~261 msg/s независимо от того, хотели ли 1k, 5k или 10k — потолок по байтам/с и стоимости операций
- Потери по метрике отчёта: 0% во всех строках.
- p95: на малых сообщениях при высокой цели растёт (десятки–сотни мс на 10k) — типичная картина нагрузки на однопоточный движок; на 100 KiB p95 остаётся умеренным, но это при низком достигнутом throughput.

**Итог:** на мелких сообщениях Redis Streams с надёжностью существенно быстрее RabbitMQ durable в этом эксперименте; на крупном payload оба сценария упираются в пропускную способность в байтах, а не в «номинальную» msg/s.

---

### 4.3. Redis Streams `inmemory_noack`

- Throughput сопоставим или выше, чем у durable на малых размерах (до ~8,5k msg/s на 128 B при 10k цели).
- p95 остаётся доли миллисекунды–единицы мс на большинстве комбинаций — режим действительно «лёгкий» по пути сообщения.
- Потери по метрике отчёта: 0% во всех строках.

**Итог:** для сценария «максимум скорости, допустим best-effort» Redis на стенде ведёт себя очень стабильно по задержкам и счётчикам доставки.

---

### 4.4. RabbitMQ `inmemory_noack` (самый неоднородный сегмент)

Здесь сочетаются реальная перегрузка, рост задержек и местами странные счётчики:

1. Потери по end-to-end счётчикам (sent vs recv): на 128 B при 5k и 10k видно ~27% недоставленного относительно `sent_count` (`recv` заметно меньше `sent`); на 10 KiB при 5k — умеренные ~6,8%; при других комбинациях — 0%. (В графике отчёта используется `lost_count`/`sent_count`; смыслы см. `runner/report.py` и `docs/decisions.md`.)

2. Задержки (p95): для in-memory RabbitMQ p95 доходит до десятков секунд на отдельных прогонах.

**Итог:** режим in-memory / no-ack у RabbitMQ на этом стенде показывает высокий пиковый throughput в отдельных ячейках матрицы, но нестабильность по доставке и задержкам при приближении к агрессивным целям

## 5. Выводы

### Какой брокер показал большую пропускную способность (msg/s)

- В режиме надёжности (`durable_ack`) Redis Streams на этом стенде стабильно выше: порядка тысяч msg/s на малых и средних сообщениях против ~200–270 msg/s у RabbitMQ — разница по порядку величины.
- В режиме «скорость важнее гарантий» (`inmemory_noack`) Redis по-прежнему даёт высокий и ровный throughput при низком p95. У RabbitMQ in-memory средние по матрице могут выглядеть высокими, но ряд прогонов нестабилен (очередь, хвосты задержек, расхождения счётчиков), поэтому сравнивать только по одной цифре throughput с Redis нельзя без контекста задержек и доставки.

### Какой брокер лучше переносит рост размера сообщения

- До ~10 KiB в сценариях с надёжностью Redis сохраняет преимущество по msg/s; RabbitMQ durable почти не меняет свой низкий потолок — он упирается не столько в размер тела, сколько в связку персистентность + ack и ресурсы одного инстанса.
- На ~100 KiB оба упираются в объём данных в секунду: у Redis durable виден явный потолок (в данных — одинаковый объём принятых сообщений за 22 с при любой целевой скорости 1k/5k/10k), то есть лимитер перестаёт быть узким местом — ограничивают Redis/диск/сериализация/один поток. Здесь сравнение «кто быстрее» сводится к байтам/с и стоимости операции, а не к номинальной msg/s.

### При какой нагрузке single instance начинает деградировать

По практике важно зафиксировать целевую скорость + размер, где растут очередь, p95, ошибки/потери:

- RabbitMQ `durable_ack`: «деградация по скорости» проявляется уже относительно низких целей (1k msg/s) — фактический throughput далёк от цели; при этом доставка по счётчикам стабильна, p95 остаётся низким — брокер не падает, а не успевает по пропускной способности.
- Redis `durable_ack`: на малых сообщениях при 5k–10k msg/s заметен рост p95 (нагрузка на один инстанс), но throughput остаётся высоким; на ~100 KiB деградация выражается в плато throughput независимо от заявленной цели.
- RabbitMQ `inmemory_noack`: деградация нелинейна — при части комбинаций огромный p95 (секунды), потери по метрикам, иногда некорректное соотношение recv/sent; для отчётности по графикам добавлен отдельный набор без этого режима (см. §3).

### Какой инструмент лучше подходит для какого сценария и почему

- Нужен брокер с явной семантикой очереди, ack и персистентности — в тесте это RabbitMQ и Redis Streams: оба подходят для постановки в очередь и гарантированной обработки.
- Если на одном инстансе важны высокая запись/чтение мелких сообщений при включённой надёжности — Redis Streams на данном стенде выигрывает по msg/s.
- Если важны предсказуемость модели, экосистема и устойчивый низкий хвост задержек при умеренной нагрузке — RabbitMQ durable остаётся сильным вариантом, но пропускную способность на таком же железе нужно отдельно планировать (не ожидать десятки тысяч msg/s без масштабирования).
- RabbitMQ in-memory / без подтверждений в этом бенчмарке показал себя как рискованный режим для сравнительных выводов; для продакшена подобный режим имеет смысл только осознанно, с отдельными SLO по потерям и задержкам.
